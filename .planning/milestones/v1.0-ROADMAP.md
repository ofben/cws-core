# Roadmap: CWS Core — Etch Job Integration

## Overview

Replace the existing 2,494-line virtual CPT approach with a clean, three-phase integration using
Etch's official `etch/dynamic_data/option` filter. Phase 1 removes the virtual CPT and establishes
the filter foundation for job listing pages. Phase 2 wires up single job page routing so
`/job/{id}/` loads a real WordPress page with injected job data. Phase 3 adds builder preview
support and polish so editors can design templates in the Etch builder with real data.

## Milestone: Dynamic Data Rebuild

**Goal:** Job data from the external API is reliably available in any Etch template via the official
`etch/dynamic_data/option` filter — works across Etch upgrades, requires no workarounds.

## Phases

- [x] **Phase 1: Core Integration** — Remove virtual CPT, add Etch filter, listing pages work

## Phase Details

### Phase 1: Core Integration

**Goal:** The virtual CPT is fully removed and `{options.cws_jobs}` is available in any Etch template.
A site editor can create a listing page in Etch using `{#loop options.cws_jobs as job}` and see
all API job fields render correctly.

**Depends on:** Nothing (first phase)

**Requirements:** Remove virtual CPT infrastructure; inject all-jobs array via `etch/dynamic_data/option`; all job API fields accessible via Etch dot notation

**Success Criteria** (what must be TRUE):
1. `class-cws-core-virtual-cpt.php` is deleted and no references to it exist in the codebase
2. An Etch template with `{#loop options.cws_jobs as job}{job.title}{/loop}` renders job titles on the frontend
3. All job fields (`title`, `company_name`, `salary`, `employment_type`, `location`, `url`, `description`, etc.) resolve correctly in Etch via dot notation
4. No PHP errors or warnings with `WP_DEBUG` enabled after virtual CPT removal
5. Existing `/job/{id}/` rewrite rules still fire (even if the full single-job template isn't built yet)

**Plans:** 2 plans

Plans:
- [x] 01-01-PLAN.md — Delete virtual CPT class, all 13 test/debug scripts, all 3 templates; strip wiring from cws-core.php, class-cws-core.php, class-cws-core-admin.php
- [x] 01-02-PLAN.md — Create class-cws-core-etch.php; add require_once to cws-core.php; filter registered, cws_jobs injected

### Phase 2: Single Job Routing

**Goal:** Visiting `/job/{id}/` renders a real WordPress page with `{options.cws_job.*}` fully
populated. The Etch template on the job page can display title, description, salary, location,
and all other job fields for the specific job in the URL.

**Depends on:** Phase 1

**Requirements:** Inject single job as `{options.cws_job}` on `/job/{id}/` URLs; load real WP page as job template; graceful 404 for unknown jobs

**Success Criteria** (what must be TRUE):
1. Visiting `/job/22026695/` (or any valid job ID) renders the configured job template page
2. `{options.cws_job.title}`, `{options.cws_job.description}`, `{options.cws_job.url}` resolve with data for that specific job
3. Visiting `/job/00000/` (invalid job) results in a 404 page, not a broken Etch template
4. If the job template WP page is missing, the admin settings page shows a visible warning
5. Changing the job slug setting automatically flushes rewrite rules

**Plans:** 3 plans

Plans:
- [x] 02-01-PLAN.md — template_redirect handler + $post/$wp_query swap + {options.cws_job} injection + 404 for unknown IDs
- [x] 02-02-PLAN.md — cws_core_job_template_page_id settings field, inline health check warning, auto rewrite flush on slug change
- [x] 02-03-PLAN.md — Gap-closure: restructure inject_options() so cws_job injection is independent of cws_jobs listing state

### Phase 3: Preview and Polish

**Goal:** A site editor can open the job template page in the Etch builder (`?etch=magic`) and see
real job data (not blank fields). Date fields are pre-formatted. Admin cache clear reliably
clears all job transients.

**Depends on:** Phase 2

**Requirements:** Etch builder preview support with fallback to configured sample job; formatted date fields; reliable cache invalidation

**Success Criteria** (what must be TRUE):
1. Opening the Etch builder on the job template page shows real job data in preview (not blank `{options.cws_job.*}` fields)
2. `{options.cws_jobs}` is also populated in the builder preview for listing template design
3. `open_date` and `update_date` are available as pre-formatted strings (e.g., `March 1, 2026`) via `{options.cws_job.open_date_formatted}`
4. "Clear Cache" in admin settings clears all job-related transients including per-job keys
5. All new code passes `WP_DEBUG` with no errors or notices

**Plans:** 2/2 plans complete

### Phase 4: Integration Cleanup

**Goal:** The Etch integration is the sole job display mechanism — no legacy content injection,
consistent data schema across listing and single-job contexts, and dead code removed.

**Depends on:** Phase 3

**Gap Closure:** Closes gaps from v1.0 milestone audit

**Requirements:** Remove legacy the_content injection; normalize listing items through format_job_data(); remove dead code

**Success Criteria** (what must be TRUE):
1. Visiting `/job/{id}/` shows only the Etch template content — no duplicate raw job block appended after it
2. `{#loop options.cws_jobs as job}{job.open_date_formatted}{/loop}` renders formatted dates on listing pages (same schema as `{options.cws_job.*}`)
3. `{#loop options.cws_jobs as job}{job.location.city}{/loop}` resolves correctly (normalized field names available on listing items)
4. No PHP errors or warnings with `WP_DEBUG` enabled
5. Dead `load_job_template()` method removed; `cws_core_job_template_page_id` default registered in activation hook
