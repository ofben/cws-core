## Unsatisfied Requirement Detail

### REQ-P2-5: Load a real WordPress page as the job detail template

**Status:** UNSATISFIED (critical blocker)

**Root cause:** `CWS_Core_Public::inject_job_data()` in `/includes/class-cws-core-public.php` is registered on the `the_content` filter at line 47 (`add_filter( 'the_content', array( $this, 'inject_job_data' ) )`). This is a pre-Phase-1 mechanism that independently fetches job data from the API and appends a raw HTML job display block (title, company, location, salary, description, apply button) to `$content` on every `/job/{id}/` page.

**Impact:** Every single-job page will render:
1. The Etch template output (correct, from the new integration)
2. **Followed by** an unstyled duplicate job block (from the legacy `inject_job_data()` method)

This conflicts directly with the milestone goal. Phase 1 deleted `class-cws-core-virtual-cpt.php` but did not modify `class-cws-core-public.php`. The `the_content` hook from the public class was never removed.

**Fix:** Remove or guard the `add_filter( 'the_content', 'inject_job_data' )` registration in `CWS_Core_Public::register_hooks()`.

## Partial Requirement Detail

### REQ-P1-3 + REQ-P3-11: Field availability on listing items

**Status:** PARTIAL (shared root cause)

**Root cause:** `CWS_Core_Etch::inject_options()` populates `$options['cws_jobs']` using `api->get_jobs()` which returns raw `queryResult` items from the API. `format_job_data()` is never called for listing items. In contrast, `$options['cws_job']` (single job) always goes through `format_job_data()`.

**Impact:**
- `{job.open_date_formatted}` and `{job.update_date_formatted}` — absent from listing items, despite being documented as available (REQ-P3-11)
- `{job.location.city}` and other normalized keys — absent from listing items (REQ-P1-3)
- Raw API field names like `{job.primary_city}` and `{job.open_date}` — do work on listing items

**Fix:** Call `format_job_data()` on each job in `inject_options()` when building the `cws_jobs` array, or document the schema difference as intentional.

## Summary

The milestone's core ambition — replacing the 2,494-line virtual CPT with a clean Etch filter integration — was successfully implemented across all three phases. The Etch integration chain (`require_once` → bootstrap → `class_exists` guard → `init()` → filter + hook registration → `inject_options()`) is fully wired and correct. Single-job routing, builder preview, date formatting, and cache clear all work correctly in code.

**Two gaps prevent milestone sign-off:**

1. **Critical:** `CWS_Core_Public::inject_job_data()` is still wired to `the_content` filter. This legacy mechanism will append a raw HTML job block after the Etch template output on every single-job page, producing visible duplicate content for real users. One file, one line to fix.

2. **Partial:** Listing items in `{options.cws_jobs}` bypass `format_job_data()` — the normalized field schema and formatted date keys available on `{options.cws_job}` are absent from listing context. Raw API field names work.

Both gaps are small and fixable. The required work is a single phase of cleanup.
